### "그럼 LLM에 물어보면 되잖아?" — 통상 RAG의 세 가지 함정  `1분`
**통상적인 RAG를 그대로 쓰면 설명 불가한 유사도 점수, 감정 텍스트 만능 자석, 폐업·중복 통과라는 함정에 빠진다 — 도구가 아니라 데이터와 구조의 문제다.**
- RAG(검색 증강 생성) :: LLM(챗GPT처럼 글을 생성하는 대형 언어 모델)이 기억으로 지어내지 않도록, 먼저 문서를 '검색'해 와서 그 근거로 답을 '생성'하는 방식 — 우리 경우 카페 후기를 찾아 읽고 추천문을 쓰는 것
- 통상 RAG의 구조 :: 문서 전체를 임베딩(문장을 숫자 좌표로 바꿔 뜻이 비슷한 말끼리 가까이 놓는 기술)해 유사도 점수로 줄 세움
- 함정 1 :: "유사도 0.83점" — 왜 이 카페가 1등인지 아무도 설명 못 한다
- 함정 2 :: "너무 좋아요" 같은 감정 문장은 아무 검색어에나 걸리는 만능 자석
- 함정 3 :: 폐업 카페도, 표기 차이로 9조각 난 카페도 점수만 높으면 그대로 추천
- 결론 :: LLM이라는 도구의 문제가 아니라, 데이터와 구조의 문제
- 🖼 *비주얼: 좌측: 통상 RAG 파이프라인 도식(문서 뭉치 → 임베딩(좌표 점들) → '유사도 0.83' 점수표 → LLM 생성). 우측: 도식의 3지점에 빨간 콜아웃 3개(점수표에 '설명 불가', 임베딩 공간에 자석 아이콘 '만능 자석', 문서 뭉치에 '폐업·중복 그대로 통과'). 하단 각주 박스에 RAG·임베딩 용어 정의 2줄.*

LLM이 기억에 의존해 지어내지 않도록, 먼저 관련 문서를 검색해 오고 그걸 근거로 답을 만들게 하는 방식이죠. 그런데 교과서적인 RAG를 카페 추천에 그대로 쓰면 함정이 셋 있습니다. 후기 전체를 임베딩, 그러니까 문장을 숫자 좌표로 바꿔 뭉뚱그린 뒤 유사도로 줄을 세우는데, '0.83점이라 1등'이라는 말은 사용자에게 아무것도 설명해 주지 못합니다. 또 '너무 좋아요' 같은 감정 문장은 아무 검색어에나 끌려 나오는 만능 자석이 됩니다. 결정적으로, 폐업한 카페도 아홉 조각으로 쪼개진 카페도 점수만 높으면 그대로 추천됩니다. 앞서 본 3대 장벽이 점수에 고스란히 스며드는 거죠. 결국 도구가 아니라 데이터와 구조를 고쳐야 하는 문제였습니다.

### 우리의 목표 세 가지 — 오늘 발표의 뼈대  `0.7분`
**하나로 묶기·살아있는 곳만·모든 문장에 출처, 이 세 목표로 신뢰를 구호가 아니라 데이터 구조로 푼다 — 이후 발표 전체가 이 뼈대를 따라간다.**
- ① 하나로 묶기 :: 표기가 달라도 같은 카페는 카드 한 장으로 (중복 해소)
- ② 살아있는 곳만 :: 폐업·휴업은 걸러내고, 실존이 검증된 카페만 추천 (실존 검증)
- ③ 모든 문장에 출처 :: 추천의 어느 문장을 짚어도 원문 근거로 거슬러 올라간다 (근거 인용)
- 신뢰를 '컨셉 문구'가 아니라 '데이터 구조'로 푼다 — 이 세 목표가 이후 데이터·검색·평가 파트의 뼈대
- 🖼 *비주얼: 가로 3장 목표 카드(하나로 묶기 / 살아있는 곳만 / 모든 문장에 출처), 각 카드에 아이콘(합쳐지는 카드 두 장 / 초록 불 켜진 간판 / 인용 부호+링크). 각 카드에서 아래로 화살표를 내려 발표 섹션명과 매핑(데이터 정본화 → 실존 검증 → 근거 인용). 우측 하단에 '신뢰 = 데이터 구조' 도장 그래픽.*

### 검색 3분기 — 질문의 종류마다 도구가 다르다  `1분`

### 시스템 구조 — 두 개의 무대: 공장과 창구  `1분`
**무겁고 실수 가능한 LLM 판단은 배치(공장)에서 끝내 사람이 검수 가능한 카드로 남기고, 런타임(창구)은 읽기만 한다 — 두 무대를 잇는 다리는 배치가 구워 둔 읽기 전용 산출물뿐이고, 그 중심이 카드 저장소다.**

- 창구는 읽기만 :: 런타임은 원본 데이터를 다시 건드리지 않는다 — 응답 1~3초(런타임 LLM 철거 전 6~9초)

### 카드 — 다섯 층으로 조립되는 읽기 전용 프로필  `1분`
**카페 한 곳 = 카드 한 장, 신원·텍스트·태그·근거·신호 다섯 층으로 쌓이며, 런타임 내내 읽기 전용 — 카드가 틀리면 서비스가 아니라 카드를 다시 굽는다.**
- ① 신원층 :: place_id · 정본 이름 · 좌표 — "이름은 흔들리지만 식별자는 흔들리지 않는다"
- ② 텍스트층 :: 검색용 자연어 요약(블로그 우선) — 다섯 층 중 임베딩에 들어가는 유일한 층
- ③ 태그층 :: 통제 사전 태그 + 지역 코드(주소에서 코드로 유도) — 필터·매칭 담당
- ④ 근거층 :: 좋았던 점 / 아쉬운 점 — 출처 달린 원문 인용. 카드 곁의 원문 스니펫 인덱스(51,661건)에서 코드가 그대로 발췌. 표시 전용, 임베딩 금지

### 런타임 흐름 — LLM은 양 끝 두 번, 몸통은 전부 코드  `1.1분`
**질문 접수부터 정렬·완화까지 검색의 몸통은 전부 읽을 수 있는 코드이고(/search는 LLM 0회), LLM은 결과를 심사하는 '판단'과 카드의 사실만 인용하는 '종합' 두 역할로만 남았다.**
- 접수(코드) :: 지역 감지(지역명 22종) + 이름 사전 1,300개 변형 조회 → 3분기: 이름 조회 / 브라우즈 / 조건 검색
- 조건 번역(임베딩) :: "석양빛"→노을 태그. 3단 게이트: 정확 매칭 → 임베딩 유사도 0.55 이상 → 미달이면 '해석 불가'(강제 번역 금지)
- 소집·검증(코드) :: 임베딩으로 후보 소집 → 어휘 근거 게이트(조건어가 카드에 실재해야 통과) → 하드 배제(노키즈존 — 어떤 완화에도 안 풀림)
- 정렬·완화(코드) :: 세부 지역 → 권역 → 전체의 완화 사다리, 완화 사실은 relaxed로 표시. 추천 이유(matched: 지역✓·겹친 태그·블로거 수)도 코드가 조립
- LLM 2회(/agent 경로) :: 판단=결과 충분/부족 심사+뺄 조건 지목(실패 시 기존 결과 유지, 무해 폴백) · 종합=카드 사실만 인용한 안내문(변호사)
- 속도 :: /search 1~3초(생성 LLM 0회) · 에이전트 경로 3~13초
- 🖼 *비주얼: 가로 흐름도. [질문] → 회색 큰 상자 '몸통 = 코드' 안에 4단계 체인: [접수: 지역 감지·이름 사전·3분기] → [조건 번역: 임베딩 3단 게이트] → [소집·검증: 근거 게이트·하드 배제] → [정렬·완화: 지역 사다리·matched 조립] → 강조색 상자 [LLM① 판단: 충분/부족 심사] (여기서 '부족'이면 조건 하나 빼고 몸통으로 되돌아가는 루프 화살표, '최대 2회' 라벨) → 강조색 상자 [LLM② 종합: 카드 사실만 인용] → [근거 답변]. LLM 상자 2개만 색을 넣고 나머지는 무채색. 하단 캡션: '/search 1~3초 (생성 LLM 0회) · /agent 3~13초'.*

<details><summary>발표 대본</summary>

마지막으로 창구의 실제 흐름입니다. 질문이 들어오면 먼저 코드가 접수합니다 — 지역을 감지하고, 이름 사전 1,300개 변형을 조회해서, 이름 조회냐 둘러보기냐 조건 검색이냐 세 갈래로 나눕니다. 조건 검색이면 임베딩이 번역기로 나섭니다. 앞 장에서 말씀드린, 뜻이 비슷한 말끼리 가까이 놓이는 성질을 써서 '석양빛'이 '노을' 태그 근처에 놓이는 것만 씁니다. 유사도 0.55를 못 넘으면 억지로 번역하지 않고 해석 불가로 남깁니다. 그다음 몸통은 전부 읽을 수 있는 코드입니다 — 후보를 모으고, 조건어가 카드에 실제로 있는지 검문하고, 노키즈존 같은 하드 배제는 어떤 완화에도 풀지 않고, 지역 완화 사다리를 타며 정렬합니다. 추천 이유까지 코드가 조립합니다. LLM은 이 몸통 밖에서 딱 두 번입니다. 하나는 판단 — 결과가 충분한지 심사하고 부족하면 뺄 조건을 지목합니다. 실패하면 기존 결과를 그대로 내는 무해한 폴백이라 LLM이 죽어도 검색은 삽니다. 다른 하나는 종합 — 변호사처럼 카드에 적힌 사실만 인용해 안내문을 씁니다. 참고로 질문을 구조로 분해하는 접수원 LLM도 모듈 구현과 시나리오 검증까지 마쳤고, 지금은 배선 대기 상태입니다. 그래서 현재 /search는 생성 LLM 0회 1~3초, 에이전트 경로도 3~13초입니다.

예상 질문: 판단 LLM이 잘못 판정하면 어떻게 되나요? / 왜 접수를 LLM 의도 분석이 아니라 코드 3분기로 하나요? / 임베딩으로 검색해 놓고 어휘 게이트를 왜 또 두나요?

</details>

### 신뢰장치 ② 하드조건 불가침 — 배제는 강등이 아니라 제거  `0.9분`
**어기면 방문 자체가 무의미한 하드조건 위반 카페는 점수를 깎는 게 아니라 후보 풀에서 제거되며, 어떤 완화에도 되살아나지 않는다.**

### 신뢰장치 ③ 완화 사다리 — 무엇을 양보했는지 라벨을 붙인 대안  `0.9분`
**0건이면 억지로 채우지 않고, 확신 낮은 소프트 조건부터 라벨을 붙여 단계적으로 넓히다가, 끝까지 없으면 없다고 말한다.**

### 신뢰장치 ④ 원문 인용 검증 — 지어낸 인용은 구조적으로 불가능  `0.8분`

- 파일럿 75건 원문 검증 실패 0건 (평가 시점 스냅샷)
- 알려진 한계(정직하게) :: '그 문장이 존재한다'는 증명이지 '맥락이 맞다'는 증명은 아님 — "조용하지 않아요"에서 '조용'만 자르는 부정어 반전은 코드 검증 밖, 문서화된 한계로 남기고 방어는 프롬프트 규칙에 기댄다
- 좋은 말만 고르지 않는다 :: 부정·혼합 리뷰도 숨김 금지 — 아쉬운 점을 나란히 표시
- 🖼 *비주얼: 가로 3단계 검문소 도식: [① 원문 문단(블로그/카카오 리뷰, 형광펜 표시된 구간)] → [② LLM 가위 아이콘: 구간 잘라냄 "창문 열면 바로 바다예요"] → [③ 코드 검문소(돋보기): 원문과 부분문자열 대조 — 위 갈래 초록 도장 '원문 그대로 → 출처와 함께 표시' / 아래 갈래 빨간 휴지통 '공백 빼고 한 글자라도 다름 → 폐기(의역·요약·짜깁기)']. 하단 왼쪽 배지 '파일럿 75건 · 검증 실패 0 (평가 시점)'. 하단 오른쪽 각주 박스(노란 테두리): '한계: 존재 증명 ≠ 맥락 증명 — 부정어 반전은 코드 검증 밖(문서화된 한계), 방어는 프롬프트 규칙'.*

<details><summary>발표 대본</summary>

네 번째, 원문 인용 검증입니다. 지어내기를 두 겹으로 막습니다. 우선 서비스 화면의 '근거 보기'는 생성이 아예 없습니다 — 코드가 블로그·리뷰 원문 스니펫을 그대로 발췌해서 보여줍니다. 그리고 LLM이 인용을 잘라내는 자리에서는 잘린 문장이 원문 안에 통째로 존재하는지 코드가 부분문자열로 대조하고, 공백 차이만 빼면 한 글자라도 다르면 폐기합니다 — 배치의 극성 분리는 이 검증으로 이미 돌았고, 답변 종합은 종합 모듈에 같은 검증을 구현해 시나리오 검증까지 마친 상태로 서버 배선(W5)을 앞두고 있습니다. 의역과 요약은 물론, 두 문장을 이어 붙인 짜깁기 인용도 통과하지 못합니다. 파일럿 75건에서 검증 실패 0건이었습니다 — 평가 시점 스냅샷 수치입니다. 다만 한계도 정직하게 말씀드리겠습니다. 이 검증은 '그 문장이 실제로 존재한다'는 증명이지 '맥락까지 맞다'는 증명은 아닙니다. '조용하지 않아요'에서 '조용'만 잘라 오는 부정어 반전은 통과할 수 있고, 이건 코드로 못 막는 알려진 한계로 문서화했습니다 — 방어는 '카드에 없는 사실 금지' 프롬프트 규칙에 기대고 있습니다. 한계가 어디에 있는지 아는 것까지가 신뢰장치라고 생각합니다.

예상 질문: 부정어 반전을 프롬프트로만 막는 게 충분한가요? 코드 레벨 대책 계획은요? / 75건이면 표본이 작지 않나요? 전량에서도 실패 0을 기대할 근거는요? / 인용의 출처가 협찬·광고 블로그면 어떻게 걸러내나요?

</details>

### 신뢰장치 ⑤ LLM 접근 통제 — 주는 쪽의 설계  `0.7분`
**환각과 홀림의 방지는 LLM의 절제에 기대지 않고, 입력에서 인기 수치·식별자를 차단하는 '주는 쪽의 설계'로 해결한다.**
- "LLM도 큰 숫자에 홀린다" :: 조회수·언급 총량 같은 인기 수치는 LLM 입력에서 차단 — 동점 정렬용으로만 사용
- 식별자(place_id·video_id·좌표)는 전부 LLM 우회, 코드가 운반 — 한 글자 환각이면 다른 가게가 된다
- 주는 것 :: 요약·태그 같은 카드 원문 파트 / 막는 것 :: 인기 수치·식별자·카드 밖 세상
- 역할 계약 :: LLM①은 접수원(질문 구조화 — router, 서버 배선 대기·현행 판단 루프는 judge가 결과 충분/부족 판정만), LLM②는 변호사(카드의 증거로만 변론) — 카드 밖 사실 금지
- 실패해도 무해 :: 판단 LLM 실패 시 기존 결과 유지 폴백 — 서비스가 LLM에 인질 잡히지 않는다
- 접근 통제는 받는 쪽의 절제가 아니라 주는 쪽의 설계다
- 🖼 *비주얼: 카드 저장소(왼쪽)에서 LLM 방(오른쪽)으로 이어지는 컨베이어 벨트 + 중간 보안 게이트 도식: 초록 게이트를 통과하는 아이템 3개(문서 아이콘 '요약', 태그 아이콘 '태그', 따옴표 카드 '카드 원문 파트(주의·반응 힌트)') / 빨간 X 게이트에 걸리는 아이템 3개(눈 아이콘 '조회수 87만', ID 카드 'place_id 1403064814', 지구본 '카드 밖 정보'). LLM 방 안 말풍선 '카드에 있는 사실만 배열'. 방 옆 작은 낙하산 아이콘 '실패 시 무해 폴백(기존 결과 유지)'. 하단 전체 폭 캐치프레이즈 배너: '접근 통제는 받는 쪽의 절제가 아니라 주는 쪽의 설계'.*

<details><summary>발표 대본</summary>

마지막 장치는 방향이 다릅니다. 앞의 넷이 출력을 검증했다면, 이건 입력을 설계합니다. 사람만 큰 숫자에 홀리는 게 아니라 LLM도 조회수가 큰 카페를 근거 없이 좋게 봅니다. 그래서 인기 수치는 LLM 입력에서 아예 차단하고 동점 정렬에만 씁니다. place_id 같은 식별자도 한 글자만 환각이 나면 다른 가게가 되므로, 전부 LLM을 우회해 코드가 운반합니다. LLM에게 건네는 건 요약·태그 같은 카드 원문 파트뿐입니다. 앞에서 쓴 비유로 정리하면, 접수원은 구조화만 하고 변호사는 증거로만 변론합니다. 정확히는, 질문 구조화 전담 접수원(router)은 시나리오 검증을 마치고 서버 배선(W5)을 기다리고 있고, 지금 서비스의 판단 루프에서는 judge가 결과의 충분·부족과 완화할 조건만 판정합니다. 그리고 판단 LLM이 실패해도 기존 결과를 유지하는 무해 폴백이라 서비스가 멈추지 않습니다. 접근 통제는 받는 쪽의 절제가 아니라 주는 쪽의 설계다 — 이 문장이 다섯 장치를 관통하는 원칙입니다. 이 장치들이 실전에서 어떤 결과를 냈는지, 이어지는 순서에서 보여드리겠습니다.

예상 질문: 인기 수치를 빼면 추천 품질이 오히려 떨어지지 않나요? 인기는 어디에 쓰이나요? / LLM 호출이 실패하면 사용자 경험은 어떻게 되나요? / 블로그 원문에 악성 지시가 들어 있는 프롬프트 인젝션은 어떻게 방어하나요?

</details>

## S6-평가  (약 4.5분)

### 3단 평가 체계 — 금·은·동으로 나눠 잰다  `0.7분`
**정확도 하나로 뭉뚱그리지 않고 Gold(사람 질문·의도), Silver(대량 자동), Bronze(시스템 건강)로 분리해야 병목이 보인다.**
- Gold 50 :: 사람이 만든 실제 질문 50개 — "대표 카페를 찾았나(Must Hit)"와 "질문 의도를 만족했나(Intent)"를 분리 채점
- Silver 168 :: 자동 생성 질의 168개 — 카페명 / 지역 / 태그 / 지역+태그 4유형별 강약 진단
- Bronze :: 시스템 건강검진 — 중복률 · 빈 결과 · 거절률 · 결정성(같은 질문에 같은 결과가 나오는 정도) · 표현 변형 안정성
- 왜 3단인가 :: "이름을 맞혔다 ≠ 좋은 추천" — 정답 적중, 의도 만족, 시스템 신뢰는 서로 다른 병을 진단한다
- 측정 환경 :: 검색 문서 1,245 · 서빙 카페 907 · Top-10 기준 — 2026-07-09 구조 개편 전 시스템의 평가 시점 스냅샷
- 🖼 *비주얼: 가로 3단 계단(메달 색 Gold/Silver/Bronze) 도식: [Gold 50 — 사람 질문 · Must Hit / Intent 분리 채점] → [Silver 168 — 자동 질의 · 카페명/지역/태그/지역+태그 4유형] → [Bronze — 건강검진 · 중복률·빈 결과·거절률·결정성·안정성]. 하단에 회색 캡션 바: "측정 환경: 검색 문서 1,245 · 서빙 카페 907 · Top-10 기준 — 2026-07-09 구조 개편 전 스냅샷".*

<details><summary>발표 대본</summary>

시스템을 만들었으니 이제 성적표를 보여드릴 차례입니다. 저희는 평가를 금·은·동 세 단으로 나눴습니다. Gold는 사람이 직접 만든 실제 질문 50개인데, 핵심은 채점을 둘로 쪼갠 겁니다 — 대표 카페를 찾았는가, 그리고 질문의 의도를 만족했는가. 이름을 맞혔다고 좋은 추천은 아니니까요. Silver는 자동 생성 질의 168개로 유형별 강약을 재고, Bronze는 정확도가 아니라 시스템 자체의 건강 — 중복은 없는지, 거절할 줄 아는지, 같은 질문에 같은 답이 나오는지를 봅니다. 한 가지 미리 말씀드리면, 지금부터 보실 수치는 7월 9일 구조 개편 전 시스템을 잰 스냅샷입니다. 이 이야기는 네 번째 슬라이드에서 다시 드리겠습니다.

예상 질문: Gold 문항은 누가 만들었나 — 카드 데이터를 본 사람이 만들면 정답이 오염되는데 어떻게 방지했나? / Bronze의 '결정성'은 구체적으로 어떻게 측정했나? / 50문항이면 통계적으로 충분한 표본인가?

</details>

### Gold 결과 — 병목은 데이터가 아니라 재정렬  `0.9분`
**필수 태그도 정답 카페도 결과 안에는 이미 있다(Coverage 85.7%, 수록률 99.6%). Top-5로 끌어올리는 재정렬이 병목이다.**
- 간극이 진단이다 :: Required Tag Coverage@5 85.7%(필수 태그가 결과 어딘가에는 있음) vs Tag Match@5 59.3%(Top-5 전체가 일관되게 만족) — 26.4%p 차이
- Must Hit @5 62% → @10 74% :: 정답 카페가 Top-10엔 들어와 있는데 Top-5로 못 올라온다
- Region Match@5 73.8% · Final Intent Score@5 65.1% · Intent High Rate@5 58.0%
- 대표 카페 DB 수록률 99.6% :: 데이터 부족이 아니다 → 데이터를 더 모으는 것은 처방이 아니다
- 실패 유형 분해(50문항) :: 완전 실패는 18% — 20%는 expected 목록 밖의 새 적합 후보를 찾은 경우(지표상 실패지만 검색 실패 아님)
- 🖼 *비주얼: 좌측: 가로 막대 그래프 5개(Tag Match@5 59.3 / Tag Coverage@5 85.7 / Region@5 73.8 / Final Intent@5 65.1 / DB 수록률 99.6). 59.3과 85.7 막대를 위아래로 붙이고 사이 간극에 "26.4%p = 재정렬 병목" 화살표 주석. 우측 상단: 2단 계단 도식 "Must Hit @5 62% → @10 74%"에 캡션 "정답은 Top-10엔 있다". 우측 하단: 50문항 유형 분해 100% 스택바 — 38% / 24% / 20%(expected 밖 새 후보) / 18%(완전 실패). 38%·24% 구간의 라벨 문구는 팀 원본 평가 시트에서 재확인 필요.*

<details><summary>발표 대본</summary>

Gold에서 가장 중요한 건 두 숫자의 간극입니다. 필수 태그가 결과 어딘가에는 있는가 — Coverage 85.7%. 그런데 Top-5 전체가 일관되게 만족하는가 — Match는 59.3%입니다. 대표 카페 적중도 같은 무늬입니다. Top-5 기준 62%가 Top-10으로 넓히면 74%가 됩니다. 정답이 시스템 안에, 심지어 10등 안에는 들어와 있는데 5등 안으로 못 올라온다는 뜻입니다. 대표 카페 수록률이 99.6%이니 데이터 부족도 아닙니다. 그래서 진단은 한 문장입니다 — 병목은 데이터가 아니라 재정렬이다. 오른쪽 유형 분해도 봐 주십시오. 50문항 중 완전 실패는 18%뿐이고, 20%는 오히려 저희 기대 목록 밖에서 새 적합 후보를 찾은 경우였습니다. 지표는 실패로 집계했지만, 실제로는 검색이 일을 한 겁니다.

예상 질문: Coverage와 Match의 정확한 계산식 차이는 무엇인가? / '재정렬 병목' 진단의 근거가 @5→@10 12%p 하나로 충분한가? / expected 밖의 새 후보가 '적합'하다는 판정은 누가 어떻게 했나?

</details>

### Silver·Bronze — 복합 의도에서 무너지고, 거절을 못 한다  `1분`
**장소를 '찾는' 능력은 안정적이지만 복합 의도를 '정렬하는' 능력이 약하고, Bronze의 뼈아픈 4개 지표가 구조 개편의 과녁이 됐다.**
- 지표 정의 :: Hit@5 = 상위 5개 안에 정답이 든 비율 · Precision@5 = 상위 5개 중 조건에 실제로 맞는 카페의 비율
- Silver 유형별 Hit@5 :: 카페명(100문항) 96.0% · 지역(13) 100% · 태그(25) 84.0% · 지역+태그(30) 76.7% — 조건이 겹칠수록 무너진다
- Precision@5는 더 가파르다 :: 지역 89.2% → 태그 55.2% → 지역+태그 40.0%
- Bronze 양호 :: Top-10 고유 카페 92.9% · 정상 질문 빈 결과 0% · 지역 메타데이터 100%

- LLM / gpt-5-mini :: LLM=대량의 글을 학습해 사람의 말·글을 읽고 쓰는 인공지능(예: ChatGPT). gpt-5-mini는 우리가 쓴 OpenAI의 경량 LLM — 오프라인 정제(996곳 요약·태그)와 런타임 /agent 경로 2회(결과 판단·근거 종합)에만 제한적으로 쓴다. 기본 검색(/search)은 LLM 호출 0회
- 임베딩 :: 문장을 숫자 좌표로 바꿔 뜻이 비슷한 말끼리 가까이 놓이게 하는 기술(모델: text-embedding-3-large). 우리는 '석양빛'이라는 말이 '노을' 태그 근처에 놓이는 성질을 두 용도로 쓴다 — 조건 검색의 후보 '소집' 그물, 그리고 사용자 언어를 태그로 옮기는 '번역기'. 카페 확정은 임베딩 점수가 아니라 조건어가 카드 근거에 실재하는지를 코드가 확인한다
- 벡터 / 벡터DB(Chroma) :: 벡터=임베딩이 만든 숫자 좌표 한 줄. 벡터DB=그 좌표들을 모아두고 '가장 가까운 것'을 빨리 찾아주는 창고. 우리는 Chroma에 카드 요약 985문서를 저장하고, 태그 번역용 앵커 93개는 별도 파일로 보관한다
- 배치 / 런타임 :: 배치=손님 오기 전에 미리 돌려두는 '공장'(수집·정제·카드 굽기 — 카드 내용을 만드는 LLM 판단은 전부 여기서 끝냄). 런타임=사용자 질문을 받는 '창구'(구워둔 카드를 읽기만 함). 카드가 틀리면 창구가 아니라 공장을 다시 돌린다
- FastAPI / 콜드스타트 :: FastAPI=파이썬으로 웹 API 서버를 만드는 도구 — /search·/agent·/evidence·/photos 창구가 전부 이 위에서 돈다. 콜드스타트=무료 배포 서버가 15분간 손님이 없으면 잠들었다가 첫 요청 때 깨어나며 생기는 대기 시간(데모 전에 미리 접속해 깨워두는 이유)
- 🖼 *비주얼: 2단 용어집 표: 왼쪽 열=용어(굵게, 예: 'RAG', '임베딩'), 오른쪽 열=정의 1문장+'우리 서비스에서는' 예시 1문장(예시 부분은 옅은 배경 하이라이트). 6행. 상단에 작은 안내 배지 '발표 중 놓친 용어는 여기로'.*

### 부록 ② 용어집 — 이 프로젝트의 언어  `0분`
**팀이 만들며 굳힌 프로젝트 고유 용어 — 이 낱말들만 알면 발표 전체의 설계 결정이 다시 읽힌다.**
- 태그 사전(통제 어휘) :: 검색에 쓰기로 약속한 공식 어휘 목록(오션뷰·노을·애견동반 등 23종 — 데이터 빈도로 승격된 포토존·핸드드립·한라산뷰 포함). 세상의 온갖 표현을 이 목록 안 단어로만 받아 적게 해서, 추천 이유를 항상 '셀 수 있는' 신호로 만든다
- 하드조건 / 소프트조건 :: 하드=절대 양보 못 하는 조건(애견동반 필수, 노키즈존 회피 등 — 어떤 완화에도 안 풀림). 소프트=아쉬우면 양보할 수 있는 취향(노을·조용함). '애견동반 필수인 사람에게 동반 불가 카페를 대안이라 주지 않는다'가 이 구분의 존재 이유
- 다수결(고유 블로거 수) :: 언급 '횟수'가 아니라 언급한 '사람 수'를 세는 정렬 신호. 한 명이 열 번 말한 카페보다 열 명이 한 번씩 말한 카페가 진짜 — 도배와 옛글에 오염되지 않는 유일한 인기 신호
- place_id / 정본화·병합 :: place_id=카카오가 장소마다 부여한 영구 번호(가게의 주민등록번호). 정본화·병합=표기 차이로 9조각 났던 '프릳츠'를 이 번호로 한 장의 정본 카드로 합치는 작업 — 이름 1,064개가 정본 카드 831장이 됐다
- 만능 자석 :: '너무 좋아요' 같은 감정 문장을 임베딩에 넣으면 아무 검색어에나 다 걸리는 현상. 그래서 반응 텍스트는 임베딩 금지, 표시층에서만 쓴다 — '층을 옮기면 독이 약이 된다'
- 완화 사다리 :: 결과가 부족할 때 한 칸씩 양보하며 다시 찾는 절차 — 지역은 세부→버킷→전체로 넓히고(서버 내장), 소프트조건은 판단 루프가 지목한 것부터 뺀다. 하드조건은 사다리 끝까지 불가침. '조용함 조건을 빼고 찾았어요' 같은 완화 문구의 화면 노출은 검증까지 끝나 서버 연결(배선) 대기
- 깔때기(funnel) :: 후보가 단계마다 줄어든 과정의 기록 — 전체 831 → 애월 87 → 노을 12 → 상위 5. 지금은 debug 모드에서 이 숫자 흐름을 재구성할 수 있고, '이렇게 찾았어요'로 사용자 화면에 노출하는 것은 설계 완료·구현 대기
- 원문 부분문자열 검증 :: 화면에 나가는 모든 인용문이 실제 원문 안에 '글자 그대로' 들어있는지 코드가 대조하는 검사. 한 글자라도 다르면 폐기 — 지어낸 인용이 구조적으로 불가능한 이유(파일럿 75건, 검증 실패 0)
- 🖼 *비주얼: 2단 용어집 표: 왼쪽 열=용어(굵게), 오른쪽 열=정의+프로젝트 예시. 8행. 4행(place_id)에는 미니 도식 '프릳츠 9조각 → 카드 1장', 7행(깔때기)에는 미니 가로 깔때기 '831→87→12→5'를 아이콘 크기로 삽입. 하단 각주: '태그 사전 23종·앵커 93개는 현재 런타임 실측 기준'.*

이 장도 부록입니다. 저희 팀이 나흘간 사건을 겪으며 만든 여덟 개의 낱말인데, 이 낱말들만 기억하시면 발표 전체가 다시 읽힙니다 — 태그 사전이 어휘를 통제하고, 하드와 소프트가 양보의 선을 긋고, 다수결이 사람 수를 세고, place_id가 신원을 보증하고, 만능 자석을 임베딩 밖으로 빼고, 완화 사다리와 깔때기가 과정을 투명하게 하고(완화 문구·깔때기의 사용자 화면 노출은 현재 구현 대기 — 질의응답에서 이 구분을 정확히 답할 것), 마지막으로 원문 검증이 모든 인용을 지켜냅니다. 질의응답 때 필요하면 이 장을 띄워두고 답하겠습니다.

### [정체성/RAG 정의·🔺송곳] 이게 RAG 맞나요? 임베딩을 번역에만 쓰면 그냥 태그 필터 검색 아닌가요?
- **한 줄 답**: 검색(Retrieval)으로 가져온 근거를 생성(Generation)에 주입하는 RAG의 정의를 그대로 충족합니다. 임베딩은 번역에'만' 쓰이는 게 아니라 두 자리에서 일합니다 — 태그 번역기(앵커 93개)와 후보 소집(카드 요약 985문서 벡터 검색). 다른 것은 부품이 아니라 각 부품의 '권한'입니다.
- **상세**: 네, RAG의 세 요소가 전부 실재합니다. 조건 검색 분기는 질의를 임베딩해 Chroma의 카드 요약 985문서에서 후보를 소집하고(Retrieval), 프론트가 쓰는 /agent 경로에서 그 카드의 근거만 인용해 LLM이 답변을 종합합니다(Augmented Generation). 태그 필터 검색과 결정적으로 다른 점은, 사전에 없는 열린 표현도 임베딩 검색이 받아낸다는 것입니다. 저희가 바꾼 것은 유사도 점수의 '권한'입니다 — 통상 RAG에서는 0.83이라는 설명 불가능한 점수가 최종 판결인데, 저희는 유사도에게 후보 소집과 게이트 통과자들 사이의 순위까지만 맡기고, 최종 자격은 '조건어가 근거에 실재하는가'를 보는 어휘 근거 게이트가 결정합니다. 브라우즈 순위는 다수결(고유 블로거 수)이고, 태그 충족 수·다수결로 조건검색 순위까지 다시 매기는 재정렬은 W2-4 retrieve 모듈에 구현돼 시나리오 검증까지 끝났으며 서빙 배선(W5)을 기다리고 있습니다. 그래서 저희 표현으로는 'RAG가 아닌 것'이 아니라 '임베딩을 찾는 도구에서 번역하는 도구로 강등시킨 RAG'입니다.

### [계약/프론트·🔺송곳] '프론트 재정렬 금지' 같은 계약을 왜 뒀나요? 프론트가 UX에 맞게 순서를 바꾸면 안 되나요?
- **한 줄 답**: 순위가 곧 추천의 실체이기 때문입니다 — 백엔드 순위는 읽을 수 있는 규칙(조회=이름 고정, 브라우즈=다수결, 조건검색=근거 게이트 통과 후보의 정렬)으로 결정되고 그 근거가 matched·번역·완화 로그로 응답에 남는데, 프론트가 재정렬하는 순간 '화면의 순위'와 '시스템이 설명하는 순위'가 어긋나 설명가능성 전체가 깨집니다.
- **상세**: 저희 컨셉이 신뢰이고, 신뢰의 구체 형태가 '이 카페가 왜 이 순서로 나왔는지 항상 설명 가능하다'입니다. 그 설명 — 어느 태그가 맞았고 블로거 몇 명이 언급했고 어떤 조건을 완화했는지 — 은 전부 백엔드가 정렬을 결정한 시점의 기록(matched·translation·relaxed)입니다. 프론트가 선의로라도 순서를 바꾸면 화면이 보여주는 1등과 시스템이 근거를 대는 1등이 달라지고, 그 순간 근거 표시 전체가 거짓말이 됩니다. 실제 사고 경험도 있습니다 — 프론트가 카페 이름 문자열로 자체 조인·중복 제거를 하던 시절, 정본명 교체로 조인 키가 어긋나는 문제를 겪고 place_id와 좌표를 직접 쓰는 방식으로 은퇴시켰습니다. 그래서 계약을 세 원칙으로 못 박았습니다 — 필드는 추가만(하위호환), 필드가 없으면 그리지 않는다, 백엔드가 이미 정렬해서 보내므로 프론트는 재정렬·중복 제거를 하지 않는다. 프론트는 순위의 소비자이지 생산자가 아니라는 역할 분리입니다.
- **근거**: FRONTEND_지침.md 3원칙(정본 계약 문서), 파도7(spot_name 조인 은퇴→place_id/lat/lng 직접 사용), matched 코드 조립=결정 29('근거가 판단을 만들지, 판단 뒤에 근거를 지어 붙이지 않는다'), 정렬 규칙=server.py:517-539(브라우즈 다수결)·:586-599(조건검색 게이트 후 정렬), 깔때기·완화 label 노출 설계(파도8 초안)

### [아키텍처/일관성·🔺송곳] '런타임 LLM 0'이라고 하셨는데 실제 웹 프론트는 LLM이 도는 /agent를 호출하지 않나요? 어느 쪽이 진짜 시스템인가요?
- **한 줄 답**: 둘 다 진짜이고 관계가 명확합니다 — LLM 0회 결정적 파이프라인인 /search가 뼈대이고, /agent는 그 /search를 '도구'로 쓰는 판단 루프(판단 최대 2회+종합 1회)입니다. /agent의 LLM은 검색 순위를 만지지 못하고, 결과가 부족할 때 완화를 제안하고 문장을 종합하는 권한만 갖습니다.
- **상세**: 좋은 지적이라 구조를 정확히 말씀드리겠습니다. 검색의 본체 — 후보 소집, 어휘 게이트, 하드 배제, 지역 완화, 순위 — 는 /search라는 LLM 0회 결정적 파이프라인이고 1~3초에 돕니다. 프론트가 호출하는 /agent는 이 /search를 내부 도구로 쓰는 얇은 판단 루프입니다 — 결과 카드를 보고 충분한지 판정하고(최대 2회), 부족하면 조건어 하나를 빼서 재검색을 시키고, 마지막에 소개 문장을 종합합니다(1회). 즉 '런타임 LLM 철거'의 정확한 의미는 LLM을 없앴다가 아니라 검색 결과를 결정하는 자리에서 뺐다는 것입니다 — 이것이 저희가 '검색엔진에 LLM을 붙인 게 아니라 LLM에게 검색 도구를 줬다'고 말하는 문장의 실체입니다. 안전장치도 그 관계를 증명합니다 — 판단 LLM이 실패하면 '충분'으로 폴백해 /search 결과가 그대로 나가고, 종합이 실패하면 소개문만 비웁니다. 시연은 /agent 경로(실측 3~13초)로 진행되며, 그 시간 차이가 곧 LLM의 자리값입니다.
- **근거**: server.py:502-630(/search LLM 0회, 주석 'LLM 층 은퇴'), :791-848(run_agent: judge 루프당 1회·최대 2루프+synthesize 1회, 실패 폴백 무해), web/index.html:697(apiSearch→GET /agent, 주석 'LLM 2~4회 실측 3~13초'), 응답 1~3초 vs 3~13초, 결정 28~30
