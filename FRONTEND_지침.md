# 프론트 지침 — /search API 계약 v2 (클로드 코드용)

> 작성: 코워크(SilenceBreaker) 2026-07-08. 백엔드 개편에 따른 프론트 대응 안내.
> 계약 원칙: **필드는 추가만 한다(하위호환). 필드가 없으면 렌더링하지 않는다. `spot_name` 조인 키는 불변.**

## 배경 (왜 바뀌나)

"해지개"처럼 카페 이름으로 검색하면 결과에 그 카페가 아예 안 나오는 문제가 실측됨
(고유명사 조회는 임베딩의 직업이 아님). 백엔드 검색이 3층으로 개편됨:

1. **이름 매치 레이어** (배포됨) — 질의에 카페명이 있으면 그 카드를 무조건 1순위 고정
2. **카드 정본(merge.py)** (작업 중) — 5개 소스 병합, 카드 필드 확장
3. **LLM 선별+근거 설명** (예정) — top 후보를 LLM이 골라 순서 매기고 "왜 이 카페인가" 생성

## 파도 1 — 이름 매치 (지금 반영 가능)

`/search` 응답 card에 추가된 필드:

```jsonc
{
  "spot_name": "해지개",
  "score": 1.0,            // 이름 매치 카드는 1.0 고정
  "name_match": true,      // ★ 신규. 질의에 이 카페 이름이 포함되어 1순위 고정됨
  ...기존 필드 동일
}
```

- `name_match: true` 카드는 배열 맨 앞에 옴 (이미 정렬돼 나감 — 프론트 재정렬 금지)
- 표시 제안: "이름 일치" 배지 또는 📌 아이콘. score 게이지를 쓰고 있다면 이 카드는 유사도가 아니라 확정 매치이므로 게이지 대신 배지가 맞음
- 이름 변형 중복(예: "해지개"와 "해지개카페")이 둘 다 PIN으로 올 수 있음 — **프론트에서 중복 제거하지 말 것.** 병합(merge.py)이 해결할 데이터 문제이고, 프론트가 가리면 문제가 안 보이게 됨

## 파도 2 — 카드 필드 확장 (merge.py 완성 후)

추가 예정 필드 (확정 시 본 문서 갱신):

```jsonc
{
  "address": "제주특별자치도 제주시 애월읍 애월북서길 52",  // 도로명 우선
  "phone": "064-...",        // 없으면 필드 자체 생략 가능
  "closed_hint": false,       // true면 폐업 의심 — 카드에 경고 표시 필요
  ...
}
```

- 렌더링 규칙 동일: **없으면 안 그린다.** mock 폴백(cards.js)에도 같은 필드를 넣을 필요 없음 — 없으면 미표시로 동작해야 함
- 좌표(`lat`/`lng`)는 기존과 동일하게 카드에 실려 나감. plotCards의 keywordSearch 임시방편은 이 좌표로 교체 가능 (조인 키 = `spot_name`, 기존 합의 유지)

## 파도 3 — LLM 선별+근거 설명 (예정, 프론트 영향 가장 큼)

응답에 추가될 것:

```jsonc
{
  "query": "...",
  "intro": "질문에 맞춰 고른 이유를 한두 문장으로...",  // ★ 신규. resultIntro 자리에
  "cards": [
    { "spot_name": "...", "reason": "노을·베이커리 후기 다수 — 질문의 '빵+노을'과 부합", ... }
  ]
}
```

- `reason` = 카드별 선정 이유(카드 데이터 근거로만 생성됨). 카드 내 1줄 표시 제안
- **지연 증가**: LLM 층이 붙으면 검색이 2~6초. 로딩 상태(스켈레톤/스피너) 필수.
  현재 mock 폴백 타임아웃 로직이 있다면 타임아웃을 8초 이상으로 상향 필요
- 카드 수가 8개 고정이 아니라 5~8개 가변이 될 수 있음
